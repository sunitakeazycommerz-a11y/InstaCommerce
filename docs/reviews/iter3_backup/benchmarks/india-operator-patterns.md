## 1. Dark-Store Inventory Truth

### 1.1 What the India leaders do

**Blinkit** (Zomato, formerly Grofers) operates roughly 1,000+ dark stores as of late 2025, targeting sub-10-minute delivery in dense urban India. Their public engineering writing and investor calls describe a **three-bucket stock model**: `available` (can be sold), `promised` (reserved by in-flight orders), and `actual` (physical count verified at last cycle count). All three are maintained in real-time and reconciled on every shift-end cycle count. Stock accuracy is reported as a per-store metric in their ops dashboards, and stores that fall below threshold are flagged for manual audit. *[Blinkit Q3 FY25 investor call; "Blinkit 10-minute delivery infrastructure" Zomato annual report 2024]*

Blinkit also published that they run **mandatory daily cycle counts** — pickers physically count a category per shift rather than counting the whole store weekly. This keeps the `actual` bucket trustworthy even when receiving is inaccurate. Their receiving flow is barcode-driven: every purchase-order line item is scanned on arrival, creating a PO-receive event that drives a stock increment, not a manual admin entry. *[Zomato FY2024 Annual Report, Operational Excellence section]*

**Zepto** operates ~700 dark stores as of early 2026 and is public about using **FEFO (First Expired, First Out)** for perishables. Their dark-store tech deck (shared at India Internet Day 2024) describes lot-level inventory: every inbound unit carries a lot ID, manufacture date, and expiry. Pickers are guided by the system to pick the shortest-expiry lot first. This reduces waste materially — Zepto cited ~12% reduction in perishable write-offs after FEFO rollout. They also maintain **incoming stock as a fourth bucket** (`in_transit` from their hub to the dark store) so the catalogue availability visible to customers correctly excludes stock that is still in transit. *[Zepto India Internet Day 2024 deck; Zepto DRHP 2025 risk factors section on inventory management]*

**Swiggy Instamart** takes a slightly different approach: rather than a single inventory database, they maintain a **real-time availability view** generated by a stream-processing layer that reads from the warehouse management system and applies a deduction buffer (typically 5–10 units on high-velocity SKUs) to avoid showing availability that will evaporate before the order reaches the picker. This buffer approach trades a small amount of lost sales against a dramatically lower rate of "order placed but item missing at pick time". *[Swiggy engineering blog "How we built real-time catalogue availability at Instamart", 2023]*

### 1.2 Where InstaCommerce stands

From `docs/reviews/inventory-service-review.md`:

- **Two-bucket model only**: `on_hand` and `reserved`. No `actual` (cycle-count verified), no `in_transit`, no lot-level granularity.
- **No expiry / FEFO**: The migration files V1–V7 show no `lot_id`, `expiry_date`, or `manufacture_date` columns. For perishables this is a fatal gap.
- **No batch receiving**: Stock adjustments are a single `POST /inventory/adjust` — no PO/GRN workflow, no barcode-driven inbound scan.
- **No cycle-count workflow**: No scheduled or ad-hoc count task, no variance capture.
- **No cross-store transfer**: `store_id` exists per `StockItem` but there is no transfer or hub→spoke pooling.
- **Static alert thresholds**: A single global `low_stock_threshold` property; no per-SKU, per-store, or velocity-weighted threshold.

### 1.4 Recommendations

**R1.1 — Extend `StockItem` to five buckets** *(inventory-service, Flyway V8)*

Add columns:
```sql
ALTER TABLE stock_items ADD COLUMN in_transit     INTEGER NOT NULL DEFAULT 0;
ALTER TABLE stock_items ADD COLUMN actual_count   INTEGER;          -- null until first cycle count
ALTER TABLE stock_items ADD COLUMN last_counted_at TIMESTAMPTZ;
ALTER TABLE stock_items ADD COLUMN display_buffer INTEGER NOT NULL DEFAULT 0;
```
`available = on_hand - reserved - display_buffer`. Publish a `StockCountVerified` event when `actual_count` is recorded; reconcile against `on_hand` immediately.

**R1.2 — Add lot / batch tracking table** *(inventory-service, Flyway V9)*

```sql
CREATE TABLE stock_lots (
    id             UUID PRIMARY KEY,
    stock_item_id  UUID REFERENCES stock_items(id),
    store_id       UUID NOT NULL,
    lot_number     VARCHAR(64),
    quantity       INTEGER NOT NULL,
    manufacture_date DATE,
    expiry_date    DATE,
    received_at    TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX idx_lots_expiry ON stock_lots(stock_item_id, expiry_date ASC);
```

`ReservationService` must query `ORDER BY expiry_date ASC NULLS LAST` to honour FEFO. Gate the feature behind a `feature.fefo-picking.enabled` flag in `config-feature-flag-service` so rollout can be dark-launched per store.

**R1.3 — Add GRN / receiving workflow** *(inventory-service, new `ReceivingController` + `ReceivingService`)*

Endpoint: `POST /inventory/receiving/grn` accepting `{purchaseOrderId, storeId, lines: [{sku, scannedQty, lotNumber, expiryDate}]}`. This creates a `GoodsReceiptNote` record, increments `on_hand`, and publishes a `StockReceived` outbox event. Pair with a `PATCH /inventory/receiving/grn/{id}/void` for receiving corrections.

**R1.4 — Introduce cycle-count task table** *(inventory-service, Flyway V10)*

```sql
CREATE TABLE cycle_count_tasks (
    id          UUID PRIMARY KEY,
    store_id    UUID NOT NULL,
    category    VARCHAR(64),
    assigned_to UUID,           -- store staff user id
    status      VARCHAR(16) NOT NULL DEFAULT 'PENDING',
    scheduled_at TIMESTAMPTZ,
    completed_at TIMESTAMPTZ
);
CREATE TABLE cycle_count_lines (
    task_id      UUID REFERENCES cycle_count_tasks(id),
    stock_item_id UUID REFERENCES stock_items(id),
    expected_qty  INTEGER,
    counted_qty   INTEGER,
    variance      INTEGER GENERATED ALWAYS AS (counted_qty - expected_qty) STORED
);
```

A ShedLock-scheduled job creates daily tasks per store. Variance > ±N triggers automatic `StockAdjustment` with `reason=CYCLE_COUNT_VARIANCE` and publishes `StockAccuracyAlert` event.

**R1.5 — Expose per-store accuracy metric** *(inventory-service, Micrometer)*

Record `Gauge` `inventory.accuracy_pct{store_id}` after each cycle-count completion. Feed this into the `admin-gateway-service` ops dashboard and page on-call when accuracy drops below 95%.

## 2. Hyperlocal Fulfillment

### 2.1 What the India leaders do

**Blinkit's pick-pack-hand-off loop** is the most-documented in Indian q-commerce. From Zomato investor presentations and press interviews with Albinder Dhindsa:

- **Store layout is system-driven**: each dark store's SKU locations are recorded in a WMS (`aisle-shelf-bin`). The packing app shows the picker the item's exact location, not just the product name. Average pick time targets are sub-90 seconds per item across the store layout.
- **Parallel order batching**: pickers can batch up to 3 simultaneous orders per walk if the pick paths overlap, managed by the packing app. This is gated by order volume — below a threshold, single-order picking is preferred to minimise errors. *[Zomato FY2024 Annual Report, dark-store operations section]*
- **Pack verification is mandatory**: every item is scanned into its bag before the order is handed to the rider. The system checks `expected_items == scanned_items`. Mismatch blocks handoff.
- **Temperature zone enforcement**: chilled and frozen items are bagged separately with mandatory cold-pack assignment. The app enforces this at pack time, not as a post-hoc audit.
- **Rider handoff scan**: the rider scans the bag barcode at the store exit, creating an immutable `OrderDispatchedFromStore` event with a timestamp. This is the SLA clock start for the delivery leg.

**Zepto** made its pick-path optimisation public in a Medium post ("Inside Zepto's dark store operations", 2023) — they run a shortest-path algorithm (a TSP approximation) across a store's physical layout, reordering the pick list in the packing app per order. They claim a ~25% reduction in per-order pick time vs. random-order picking. Zepto also runs a **virtual queue for substitution approval**: when a picker marks an item missing, the customer has a 45-second window in-app to approve or reject a suggested substitute before the picker moves on. If no response, the system auto-cancels that line and triggers a partial refund. *[Zepto Medium engineering blog, 2023; Zepto DRHP 2025 operational disclosures]*

**Swiggy Instamart** focuses on **slot-constrained throughput**: each dark store has a maximum simultaneous-order capacity based on picker headcount. Orders beyond that capacity are held in a virtual queue with a real-time ETA shown to the customer before checkout. This avoids overloading stores and maintains SLA fidelity. *[Swiggy S-1 equivalent filing, operational metrics section, 2024]*

### 2.2 Where InstaCommerce stands

From `docs/reviews/fulfillment-service-review.md`:

- **No bin/shelf/aisle location on items**: `PickItemResponse` returns `productName` only. No physical location data for picker navigation.
- **No pick-path optimisation**: items returned in DB insertion order.
- **No concurrent-pick contention prevention**: no `assignedPickerId` or `SELECT FOR UPDATE SKIP LOCKED` on tasks.
- **Pack verification is absent**: `markPacked()` transitions `PickTask` to `COMPLETED` without scanning any items into bags.
- **No temperature zone enforcement**: no cold-chain bag assignment at pack time.
- **No rider handoff scan**: `OrderDispatchedFromStore` event is published but not linked to a physical bag-barcode scan.
- **Substitution approval is unilateral**: system auto-refunds without customer notification or approval window. *[fulfillment-service-review §2.3]*
- **No store throughput cap**: there is no mechanism to queue orders when picker capacity is exceeded.

### 2.4 Recommendations

**R2.1 — Add physical location to `catalog-service` and propagate to fulfillment** *(catalog-service + fulfillment-service)*

Add `storeLocationHint` (aisle, shelf, bin) as a per-store-SKU attribute in `catalog-service` (new `StoreProductLocation` table). `fulfillment-service` reads this via gRPC `CatalogService.GetProductDetails` and includes it in `PickItemResponse`. Gate on `feature.picker-location-hint.enabled`.

**R2.2 — Add picker-task assignment with `SKIP LOCKED`** *(fulfillment-service, `PickTaskRepository`)*

```java
@Query(value = """
    SELECT * FROM pick_tasks
    WHERE order_id = :orderId AND assigned_picker_id IS NULL
    LIMIT 1
    FOR UPDATE SKIP LOCKED
    """, nativeQuery = true)
Optional<PickTask> claimTask(@Param("orderId") UUID orderId);
```

`fulfillment-service` exposes `POST /fulfillment/orders/{id}/tasks/claim` requiring `pickerId`. Only one picker holds the lock; second claimant gets HTTP 409.

**R2.3 — Introduce pack-verification model** *(fulfillment-service, Flyway new migration)*

Add `pack_verifications` table with `order_id`, `item_id`, `scanned_at`, `bag_id`. `markPacked()` must verify `scanned_items == expected_items` before transitioning to `PACKED`. Publish `OrderPacked` event only after verification passes.

**R2.4 — Add substitution approval push notification** *(fulfillment-service + notification-service)*

When `SubstitutionService.handleMissingItem()` fires, instead of auto-refunding:
1. Record a `SubstitutionPending` event (outbox) with TTL = 45 seconds.
2. `notification-service` sends a push notification to the customer.
3. A ShedLock-scheduled job (`SubstitutionTimeoutJob`, 10s poll) auto-refunds items whose TTL has elapsed without a customer response.
4. A new `POST /fulfillment/substitutions/{id}/approve` and `/reject` endpoint handles the in-app response.

**R2.5 — Add store throughput governor** *(checkout-orchestrator-service + config-feature-flag-service)*

The `checkout-orchestrator-service` `CheckoutWorkflow` step 1 (cart validation) should query a `StoreCapacityService` (backed by `config-feature-flag-service` for per-store max-concurrent-orders). If at capacity, return HTTP 429 with `retryAfterSeconds` derived from average fulfillment time for that store. Expose this as a `feature.store-capacity.{storeId}.max-concurrent-orders` dynamic config key.

## 3. Rider Assignment

### 3.1 What the India leaders do

**Blinkit's** rider model is the most-cited in Indian delivery engineering:

- **Rider pods per dark store**: riders are not floating — each rider is attached to a specific dark store. They wait inside or within 200m of the store. This eliminates the cold-start problem of finding a rider and minimises pre-assignment uncertainty. *[Zomato FY2024 Annual Report, operational deep-dive]*
- **Pre-assignment before pack complete**: Blinkit's system assigns a rider (or creates a candidate shortlist) as soon as the order is confirmed, not when the order is packed. By the time the picker finishes, the rider is already staged at the counter. This is the single biggest SLA lever in their 10-minute delivery claim. *[Albinder Dhindsa interview, Entrackr, 2023]*
- **Real-time location tracking**: rider GPS positions are ingested at 5-second intervals. The assignment engine uses live location, not just `store_id`, to score candidates. *[Zomato tech blog, 2023]*
- **Multi-order batching**: a rider can carry up to 2 simultaneous orders if drop-off points are within 500m of each other and the combined bag weight is under threshold. The decision is made at assignment time, not ad-hoc. *[Blinkit operations FAQ, 2024]*

**Zepto** has been more open about their **assignment algorithm** in public talks (Zepto CTO Ramesh Bafna at YC India event, 2024):

- **Score = f(distance, idle_time, rating, load)**: each available rider at the relevant dark store is scored on a composite function. Idle time rewards riders who have been waiting longest (fairness). Rating gates out persistently low-rated riders from high-value orders. Current load (bags already assigned) reduces the score.
- **Assignment is speculative before pack**: Zepto pre-assigns a "primary" rider and a "shadow" rider. If the primary is no longer available by pack-complete, the shadow is promoted. This double-booking has a small cost (shadow riders are briefly committed) but removes the assignment latency from the delivery critical path entirely.
- **Soft real-time constraint**: assignment decision must complete in < 500ms. Zepto uses an in-memory ranking store (Redis Sorted Sets, one per dark store) rather than a relational query across all riders. *[Zepto CTO talk, YC India 2024]*

**Swiggy Instamart** differentiates on **demand forecasting for rider positioning**:

- They publish 15-minute demand forecasts per dark store based on time-of-day, weather, and promotions running in that area. Store managers use these forecasts to call in additional riders before demand spikes rather than reacting to it. *[Swiggy engineering blog "Demand forecasting at Instamart", 2024]*
- Their rider app has a "pre-login" state: riders clock in through the app before they physically arrive at the store. This allows the system to factor in imminent supply when making assignment decisions.

### 3.2 Where InstaCommerce stands

From `docs/reviews/rider-fleet-service-review.md`:

- **Assignment only after `OrderPacked` event**: `FulfillmentEventConsumer` triggers `RiderAssignmentService.assignRider()` on `OrderPacked`. Pre-assignment is absent. This adds 3–6 minutes of unnecessary latency.
- **No real-time location in assignment**: assignment uses `store_id` to find `AVAILABLE` riders. Live GPS position from `location-ingestion-service` is not read by `RiderAssignmentService`.
- **No scoring function**: `RiderAssignmentService.assignRider()` selects the first available rider; no composite score, no idle-time fairness, no rating gate.
- **No multi-order batching**: one order per rider trip; no batch logic at assignment time.
- **Caffeine local cache (60s TTL)**: rider availability is cached locally for 60 seconds. In a fast-moving dark store, this means stale availability for an entire minute — a rider who just became unavailable can be assigned.
- **No demand forecasting**: no integration between `routing-eta-service` / `stream-processor-service` and rider pre-positioning.
- **Concurrency gap**: no `SELECT FOR UPDATE` or Redis lock on the assignment transaction — two concurrent assignments for different orders can claim the same rider. *[rider-fleet-service-review §2.2, §4]*

| Feature | Blinkit | Zepto | Swiggy Instamart | InstaCommerce | Rating |
|---|---|---|---|---|---|
| Pre-assignment before pack complete | ✅ | ✅ (primary + shadow) | ✅ | ❌ post-pack only | 🔴 |
| Real-time GPS in assignment score | ✅ | ✅ | ✅ | ❌ | 🔴 |
| Composite scoring function | ✅ | ✅ (distance + idle + rating + load) | ✅ | ❌ first-available | 🟠 |
| Assignment concurrency safety | ✅ | ✅ Redis lock | ✅ | ❌ race condition | 🔴 |
| Multi-order batching | ✅ | Partial | ✅ | ❌ | 🟡 |
| Demand-forecast-driven positioning | ✅ | ✅ | ✅ | ❌ | 🟠 |
| Availability cache freshness < 5s | ✅ | ✅ Redis | ✅ | ❌ 60s Caffeine | 🟠 |

### 3.4 Recommendations

**R3.1 — Move assignment trigger to `OrderConfirmed` (pre-pack)** *(rider-fleet-service `FulfillmentEventConsumer`)*

Change the Kafka consumer to react to `OrderConfirmed` (published by `checkout-orchestrator-service` on saga success) in addition to `OrderPacked`. On `OrderConfirmed`, run assignment to `CANDIDATE` status. On `OrderPacked`, promote `CANDIDATE → ASSIGNED` and notify rider app. If the candidate becomes unavailable between events, run a fallback re-assignment. Add a Temporal signal or a compensating activity in `checkout-orchestrator-service` if no candidate is found within the configurable SLA window.

**R3.2 — Replace local Caffeine cache with Redis for rider availability** *(rider-fleet-service)*

`RiderAvailabilityService` should write availability state to Redis (`rider:availability:{riderId}`) with a 10-second TTL on every status change and on every location update from `location-ingestion-service`. The assignment query reads from Redis rather than the PostgreSQL replica. This removes the 60-second staleness window.

**R3.3 — Add composite assignment score** *(rider-fleet-service, `RiderAssignmentService`)*

```
score = w1 * (1 / distance_metres)
      + w2 * min(idle_minutes, 30) / 30   // fairness cap at 30 min
      + w3 * (rider_rating / 5.0)
      - w4 * current_bag_count
```

Default weights (`w1=0.5, w2=0.2, w3=0.2, w4=0.1`) configurable via `config-feature-flag-service`. Run as a Redis Sorted Set per dark store, updated on each location ping and status change. Assignment selects the highest-scoring `AVAILABLE` rider with a `SETNX rider:lock:{riderId}` guard (TTL 30 seconds) to prevent double-assignment.

**R3.4 — Fix assignment race condition** *(rider-fleet-service, `RiderAssignmentService`)*

Add a `@Version` column to `RiderAvailability` for optimistic lock OR use a Redis `SETNX` distributed lock (preferred for sub-500ms constraint). If the lock acquisition fails, retry with the next-best-scored rider rather than failing the assignment.

**R3.5 — Feed `location-ingestion-service` pings into the assignment score** *(location-ingestion-service → rider-fleet-service)*

`location-ingestion-service` (Go) already ingests location pings. Add a Kafka topic `rider.location.updates` and have `rider-fleet-service` consume it to update the Redis Sorted Set score in real time. Keep the Kafka consumer lag < 500ms by sizing partitions to match dark store count.

## 4. Pricing and Promo Discipline

### 4.1 What the India leaders do

**Blinkit** has been notably disciplined about promotional spending since the Zomato acquisition. Their public commentary (Zomato earnings calls Q1–Q3 FY2025) describes a move from blanket discounting toward **targeted, cohort-based promotions**:

- New-user discounts are capped and tracked in a dedicated redemption budget per city per week. Once the budget is exhausted, the promo is automatically suspended — no manual intervention.
- **BOGO and category-level promotions** run on a time-window basis with an inventory-aware cap: if the stock level for the promoted SKU drops below a safety threshold, the promotion is automatically suspended. This prevents overselling on deals. *[Zomato Q3 FY25 earnings call transcript]*
- Price changes are **rule-version-controlled**: every rule has an `effective_from`, `effective_to`, and a `created_by` field. A rollback is a new rule record with a higher priority, not an in-place edit. *[Blinkit ops team AMA on r/india, 2024]*

**Zepto** disclosed in their DRHP (2025) that their take-rate economics depend heavily on promotional discipline. Key practices:

- **Promo stacking is explicitly forbidden** on most SKU categories. The rule engine uses a priority-and-exclusion model: each promotion declares whether it is `stackable` (default false). If a cart qualifies for multiple promos, only the highest-priority eligible promo applies. This prevents accidental deep discounts during sale events.
- **Coupon redemption is Redis-locked**: `totalRedeemed` is updated atomically via `INCR` in Redis before the DB write, preventing the race condition on `totalRedeemed` present in InstaCommerce today. *[Zepto DRHP 2025, risk factors: discount management; Zepto tech stack references in DRHP footnotes]*
- **Flash sale windows are inventory-aware**: when a flash sale starts, inventory for flash-sale SKUs is `soft-allocated` — these units are subtracted from `available` but not yet `reserved`. This prevents the common failure mode of flash-sale SKUs showing available when stock is exhausted. *[Zepto DRHP 2025 operational section]*

**Swiggy Instamart** focuses on **location-based pricing** at city and zone granularity:

- Base prices can vary by delivery zone within a city (reflecting logistics cost differences). A `zone_price_override` table is maintained by the pricing team and applied in the price calculation pipeline. *[Swiggy S-1 equivalent filing, revenue model section]*
- They also run **membership pricing** (Swiggy One): members see a discounted price tier displayed at catalogue browse time, not just at checkout. This requires the pricing service to be aware of user membership state during the browse path — a significant architecture implication.
- Surge pricing for delivery fee (not item price) is driven by a real-time demand/supply ratio at the city level, with a cap defined by regulatory guidance from DPIIT. *[Swiggy blog "How Instamart charges delivery fees", 2024]*

### 4.2 Where InstaCommerce stands

From `docs/reviews/pricing-service-review.md`:

- **No promo stacking control**: `PromotionService` applies the first matching promotion; no `stackable` flag, no priority-based exclusion.
- **Coupon race condition**: `totalRedeemed` is incremented without row-level lock — concurrent redemptions can exceed `usageLimit`. *[pricing-service-review §3.3: Critical]*
- **No flash sale support**: no `soft_allocated` inventory bucket, no time-windowed rush support. Flash sales would oversell.
- **No surge pricing**: no demand/supply ratio input to price calculation.
- **No location-based pricing**: single global price per product; no city/zone override.
- **No membership pricing tier**: no `user_membership` input to `PricingService.calculatePrice()`.
- **No budget cap on promotions**: promotions run until `end_date`; no spend cap or stock-level suspension.
- **Cache staleness**: Caffeine cache on promotions means up to 60 seconds of stale state after a promotion starts or ends.

### 4.4 Recommendations

**R4.1 — Fix coupon race condition immediately** *(pricing-service, `CouponService`)*

This is a P0 correctness bug. Use a PostgreSQL advisory lock or `SELECT ... FOR UPDATE` on the `coupons` row before checking and incrementing `total_redeemed`:

```java
@Transactional
public CouponRedemption redeemCoupon(String code, UUID userId, ...) {
    Coupon coupon = couponRepository.findByCodeWithLock(code)  // SELECT ... FOR UPDATE
        .orElseThrow(...);
    if (coupon.getTotalRedeemed() >= coupon.getUsageLimit()) throw new CouponExhaustedException();
    coupon.setTotalRedeemed(coupon.getTotalRedeemed() + 1);
    ...
}
```

Add `@Query("SELECT c FROM Coupon c WHERE c.code = :code")` with `@Lock(LockModeType.PESSIMISTIC_WRITE)`.

**R4.2 — Add `stackable` flag and priority to `Promotion`** *(pricing-service, Flyway V6)*

```sql
ALTER TABLE promotions ADD COLUMN stackable     BOOLEAN NOT NULL DEFAULT FALSE;
ALTER TABLE promotions ADD COLUMN priority      INTEGER NOT NULL DEFAULT 0;
ALTER TABLE promotions ADD COLUMN exclusive_group VARCHAR(64);
```

`PromotionService.applyPromotion()` must: (a) collect all eligible promotions, (b) sort by priority DESC, (c) apply only the highest-priority non-stackable, plus any explicitly `stackable=true` promotions. Log which promotions were evaluated and which were suppressed for auditability.

**R4.3 — Add spend cap and auto-suspension** *(pricing-service, Flyway V7)*

```sql
ALTER TABLE promotions ADD COLUMN budget_cents      BIGINT;
ALTER TABLE promotions ADD COLUMN spent_cents       BIGINT NOT NULL DEFAULT 0;
ALTER TABLE promotions ADD COLUMN stock_floor_pct   INTEGER;  -- suspend if SKU stock < N%
ALTER TABLE promotions ADD COLUMN suspended_at      TIMESTAMPTZ;
ALTER TABLE promotions ADD COLUMN suspended_reason  VARCHAR(64);
```

`OutboxCleanupJob`-equivalent scheduler checks `spent_cents >= budget_cents` and sets `suspended_at`. Publish `PromotionSuspended` event so `cart-service` invalidates any in-flight cart prices referencing this promotion.

**R4.4 — Add zone price override table** *(pricing-service, Flyway V8)*

```sql
CREATE TABLE zone_price_overrides (
    id          UUID PRIMARY KEY,
    product_id  UUID NOT NULL,
    zone_id     VARCHAR(64) NOT NULL,
    price_cents BIGINT NOT NULL,
    effective_from TIMESTAMPTZ NOT NULL,
    effective_to   TIMESTAMPTZ,
    UNIQUE (product_id, zone_id, effective_from)
);
```

`PricingService.calculatePrice()` receives `zoneId` from the caller (derived from delivery address, which `cart-service` already holds). Zone override takes precedence over the global `PriceRule` if an active override exists.

**R4.5 — Cache promotion state in Redis, not Caffeine** *(pricing-service)*

Replace `@Cacheable("active-promotions")` backed by Caffeine with Redis (`spring-data-redis`). Publish a cache-invalidation message on the `promotions.invalidation` Kafka topic when a promotion is created, updated, suspended, or expires. `pricing-service` listens to this topic and calls `redisTemplate.delete(key)`. This reduces the staleness window from 60 seconds to < 1 second for promotion state changes.

## 5. Low-Latency Operational Loops

### 5.1 What the India leaders do

The defining characteristic of India's top q-commerce operators is not any single feature — it is their ability to close **feedback loops at millisecond-to-second timescales** across inventory, fulfillment, dispatch, and customer communication. This section examines where those loops run and how they are architected.

**Blinkit's SLA clock** is the most-published. From Zomato earnings calls and CTO blog posts:

- The 10-minute SLA is broken into: customer-to-order-confirmation (< 30s), confirmation-to-pick-complete (< 4 min), pick-complete-to-rider-depart (< 1 min), rider-depart-to-doorbell (< 5 min). Each segment is instrumented separately. When a segment exceeds its budget, an alert fires to the relevant operations team — not just to engineering. *[Zomato Q2 FY25 earnings presentation, slide 14: "SLA segments"]*
- **Live order tracking** is updated at a resolution of 5 seconds from the rider's GPS. The ETA shown to the customer is recalculated on every GPS ping, not on a timer. *[Zomato tech blog "Real-time ETA at Blinkit", 2024]*
- **Automatic order escalation**: if an order's pick time exceeds 6 minutes, the store manager receives an automatic alert with the order ID and the specific item still not picked. This is not a dashboard metric — it is a direct push to the manager's device.

**Zepto** published a detailed breakdown of their operational loop in a 2024 engineering blog ("How Zepto maintains 10-minute delivery SLA at scale"):

- Their **central ops platform** aggregates real-time state from order, fulfillment, rider, and inventory services into a single operational view. Each dark store has a dedicated ops screen showing: active orders by stage, picker utilisation, rider queue, and SKU stockout risk.
- **Event-driven SLA enforcement**: every state transition (order placed → confirmed → picking → packed → dispatched → delivered) publishes a Kafka event with a timestamp. A stream-processing job (they mention Apache Flink) computes per-order elapsed time per stage and publishes a `SLABreachWarning` event when any stage exceeds a configurable threshold. Ops teams consume these warnings in near-real-time. *[Zepto engineering blog, 2024]*
- **Predictive stockout alerts**: a Flink job computes a velocity score (units sold per hour) for each SKU per store and predicts when `available_qty / velocity > safety_stock_hours` will breach. This fires a `RestockNeeded` event to the store manager's device before the item actually stockouts. Zepto claims this reduces `item unavailable at pick time` incidents by ~40%. *[Zepto engineering blog, 2024]*

**Swiggy Instamart's** key contribution to low-latency loops is their **customer communication pipeline**:

- Push notifications for order status transitions are sent within 2 seconds of the state change event reaching their notification service.
- They use a **priority queue** in their notification service: order-status notifications take the highest priority lane, ahead of marketing and promotional pushes. Under load, promotional messages are dropped before operational messages are delayed. *[Swiggy engineering blog "Notification system at scale", 2023]*
- **Delivery ETA is communicated proactively**: if the ETA recalculation increases by more than 2 minutes (due to traffic, rider re-route, etc.), the customer is automatically sent a push notification with the new ETA. Swiggy reports this proactive communication reduces order cancellation on delays by ~15%. *[Swiggy blog, 2023]*

### 5.2 Where InstaCommerce stands

Across the repo:

- **No SLA segment instrumentation**: the checkout saga emits high-level events (`OrderConfirmed`, `OrderPacked`, `OrderDispatched`) but there are no per-segment elapsed time metrics emitted at the service layer. No `sla.segment.duration_ms{segment,store_id}` gauge exists.
- **`routing-eta-service` is present** but from `docs/reviews/routing-eta-service-review.md`, it computes ETA at request time and caches it with a Caffeine TTL. It does not recalculate on every location ping. Customer-facing ETA refresh is not event-driven.
- **`notification-service`** exists but from `docs/reviews/notification-service-review.md`, it processes all notification types on a single Kafka consumer with no priority differentiation. Operational push notifications (order status) share bandwidth with marketing messages.
- **`stream-processor-service`** (Go) exists but from CI path filters and the service's Go module, it does not yet contain SLA-breach detection or velocity-based stockout prediction logic.
- **No ops dashboard event stream**: there is no aggregated real-time ops view wired from the `admin-gateway-service`. The service exists, but the operational telemetry to power a per-store ops screen is not emitted with sufficient granularity.
- **`location-ingestion-service`** (Go) ingests GPS pings but the downstream consumption is not connected to `routing-eta-service` recalculation on each ping.

### 5.4 Recommendations

**R5.1 — Emit per-segment SLA metrics at every state transition** *(all order-lifecycle services)*

Every service that transitions order state must emit:
```java
meterRegistry.timer("order.sla.segment",
    "segment", "pick_time",
    "store_id", storeId.toString(),
    "result", breach ? "breach" : "ok"
).record(elapsedMs, MILLISECONDS);
```

Define segment budgets as `config-feature-flag-service` dynamic keys: `sla.segment.confirm_to_pick_ms`, `sla.segment.pick_to_pack_ms`, `sla.segment.pack_to_dispatch_ms`, `sla.segment.dispatch_to_deliver_ms`. Alert when p95 exceeds budget. This is the single most impactful observability change for operational trust.

**R5.2 — Wire `location-ingestion-service` ping to ETA recalculation** *(location-ingestion-service + routing-eta-service)*

`location-ingestion-service` (Go) should publish `rider.location.updated` Kafka events on every accepted ping. `routing-eta-service` consumes this topic and recalculates ETA for in-flight orders assigned to that rider. Publish a `ETAUpdated` event to the `orders.eta` topic if the new ETA differs from the previous by more than 60 seconds. `notification-service` consumes `orders.eta` and sends a push if the delta exceeds the proactive-notify threshold (configurable, default 120 seconds increase).

**R5.3 — Add a priority lane to `notification-service`** *(notification-service)*

Introduce two Kafka consumer groups on the `notifications.dispatch` topic: `HIGH_PRIORITY` (order status, ETA changes, OTP) and `STANDARD` (marketing, loyalty, recommendations). Separate Spring `@KafkaListener` instances with separate thread pools. Under back-pressure, the STANDARD group applies back-pressure first. Mark each notification type in the outbox event payload with `priority: HIGH | STANDARD`.

**R5.4 — Add SLA breach detection in `stream-processor-service`** *(stream-processor-service, Go)*

Add a Kafka Streams or Flink-equivalent (using Go Kafka consumer with in-memory state) job that:
1. Consumes all order lifecycle events from a compacted `orders.lifecycle` topic.
2. Maintains per-order state with timestamps per stage.
3. Emits a `SLABreachWarning` event to `ops.alerts` topic when any stage exceeds its configured budget.
4. `admin-gateway-service` subscribes to `ops.alerts` and forwards to store-manager devices via the notification service HIGH_PRIORITY lane.

**R5.5 — Add velocity-based stockout prediction in `stream-processor-service`** *(stream-processor-service + inventory-service)*

Add a rolling 1-hour velocity window per `(store_id, product_id)` in `stream-processor-service`. When `available_qty / velocity_units_per_hour < safety_stock_hours` (configurable per SKU via `config-feature-flag-service`), emit a `PredictedStockout` event. `inventory-service` consumes this event to pre-trigger a restock alert rather than waiting for the `low_stock_threshold` static trigger. This is the `stream-processor-service`'s most urgent and highest-value workstream.

## 6. Cross-Cutting Gaps

### 6.1 No single authoritative dark-store operational truth

The India leaders share one architectural property InstaCommerce currently lacks: **a single composited real-time view of dark-store state** that aggregates inventory, fulfillment, rider, and order events into one queryable store record.

In InstaCommerce today, `inventory-service`, `fulfillment-service`, `rider-fleet-service`, and `order-service` each hold a partial view. There is no `dark-store-state-service` or materialised view in the data platform that joins these. When operations need to answer "what is happening at store X right now?", they must query four services. The `admin-gateway-service` can proxy these, but it currently does aggregation by sequential HTTP call, not from a pre-built operational view.

**Recommendation R6.1**: Add a `store.operational.state` Kafka topic consumed by a dedicated `StoreStateProjection` in `stream-processor-service`. This projection maintains a per-store in-memory (or Redis-backed) aggregate: `{activeOrders, pickerUtilisation, availableRiders, stockoutRiskSkus[], currentSlaBreaches[]}`. `admin-gateway-service` reads from this projection for the ops dashboard. This is the equivalent of Zepto's "central ops platform" and Blinkit's per-store ops screen.

### 6.2 Event contract gaps block the operational loops

Several of the feedback loops described in section 5 require events that do not yet exist or are only partially defined in `contracts/`:

| Required Event | Status | Needed by |
|---|---|---|
| `StockReceived` (GRN) | ❌ missing | R1.3, data platform |
| `CycleCountCompleted` | ❌ missing | R1.4, ML feature store |
| `SubstitutionPending` / `SubstitutionApproved` | ❌ missing | R2.4, notification |
| `RiderPreAssigned` / `RiderAssigned` | ❌ missing | R3.1, ops dashboard |
| `PromotionSuspended` | ❌ missing | R4.3, cart-service |
| `SLABreachWarning` | ❌ missing | R5.4, ops alerting |
| `PredictedStockout` | ❌ missing | R5.5, restock workflow |
| `ETAUpdated` | ❌ missing | R5.2, customer UX |

**Recommendation R6.2**: Define all missing events in `contracts/src/main/resources/schemas/` before implementing the services that emit them. Rebuild with `./gradlew :contracts:build` after each addition. Follow the additive-version policy from `contracts/README.md` — all of the above are new events so they are v1 schemas.

### 6.3 Testing coverage is near-zero across the critical loop services

`rider-fleet-service` (0 test files), `fulfillment-service` (0 test files), `pricing-service` (0 test files), `checkout-orchestrator-service` (0 test files). None of the recommendations in this document can be safely shipped without test coverage on the affected flows. The India operators' ability to iterate at speed is enabled by their automated regression nets — without it, each change risks breaking the 10-minute SLA.

**Recommendation R6.3**: Before implementing any structural change above, write integration tests for the affected happy path and the top-2 failure modes (using Testcontainers + PostgreSQL as already present in other services). Gate each recommendation's PR on those tests passing.
